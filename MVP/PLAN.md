﻿﻿﻿﻿﻿# План реализации расширения обработки банковской выписки

Дата составления: 2026-04-28

## Текущий статус

Статус реализации: **исходники расширены до правил заполнения реквизитов банковских документов; требуется перенос метаданных в расширение 1С и проверка в базе**.

Уже выполнено:

- Обсуждены требования к первой версии.
- Изучен общий модуль типовой конфигурации `ЗагрузкаВыпискиПоБанковскомуСчету`.
- Найдены ключевые точки типовой логики:
  - `РаспознатьВыписки()` - формирует `РаспознаннаяОперация.ХозяйственнаяОперация`.
  - `НайтиСчетНаОплатуПокупателю()` - типовой поиск неоплаченных счетов покупателя.
  - `ЗаполнитьДаннымиБанкаРаспознаннаяОперация()` - переносит `СчетНаОплату` в строку `РасшифровкаПлатежа`.
  - `СоздатьДокументы()` - создает и записывает банковские документы.
- Проверен пример файла выписки `kl_to_1c.txt`; в нем есть входящие оплаты с назначением платежа по счетам.
- Создан комплект исходников в каталоге `src/`:
  - описание метаданных расширения;
  - серверный общий модуль правил;
  - фрагмент расширения типового общего модуля;
  - сценарий проверки.

## Зафиксированные решения

- Реализация выполняется расширением конфигурации для 1С:Бухгалтерия 3.0.194.14.
- Режим расширения: **Изменение и контроль поведения**.
- Первая версия работает при автоматическом создании новых документов из файловой выписки и DirectBank.
- Первая версия обрабатывает новые поступления и списания, создаваемые загрузкой банковской выписки.
- Для входящих оплат покупателей отдельно поддержан `СчетНаОплату` в табличной части `РасшифровкаПлатежа`.
- Правила настраиваются пользователем в 1С, а не зашиваются в код.
- Если подходит несколько правил, применяется первое подходящее правило по приоритету.
- Результат применения правила фиксируется в отдельном регистре сведений/журнале расширения.
- Автоматическое распределение одного платежа по нескольким счетам не входит в MVP, но должно быть заложено как будущий режим.
- Правило может заполнять не только `СчетНаОплату`, но и вид операции, контрагента, договор, банковский счет, статью ДДС, счета Дт/Кт и статью прочих доходов/расходов.
- Для каждого заполняемого реквизита используется отдельное действие: `НеИзменять`, `Заполнить`, `Очистить`. Это позволяет отличать пустое значение от команды очистки.

## MVP

### Настройка правил

Создать пользовательский объект настройки правил обработки выписки.

Предлагаемая модель:

- `Справочник.ПравилаОбработкиБанковскойВыписки` или регистр сведений с формой списка и формой элемента.
- Реквизиты правила:
  - `Активно`
  - `Приоритет`
  - `Наименование`
  - `Организация` - необязательный ограничитель
  - `БанковскийСчетОрганизации` - необязательный ограничитель
  - `ВидОперацииДокумента` - необязательный ограничитель
  - `Контрагент` - необязательный ограничитель
  - `ИННПлательщика` - необязательный ограничитель
  - `РежимПоискаСчета`
  - `РежимЗаполненияСчета`
- `СчетНаОплатуПокупателю` - используется при режиме конкретного счета
- `НаправлениеОперации`
- `ВидОперацииПоступление`
- `ВидОперацииСписание`
- `ЗаполняемыйКонтрагент`
- `ЗаполняемыйДоговорКонтрагента`
- `ЗаполняемыйБанковскийСчетКонтрагента`
- `СтатьяДвиженияДенежныхСредств`
- `СчетДт`
- `СчетКт`
- `ПодразделениеДт`
- `ПодразделениеКт`
- `СтатьяПрочихДоходовИРасходов`
- Для каждого заполняемого реквизита добавить соседний реквизит действия `Действие...` типа `ПеречислениеСсылка.ВП_ВидыДействийЗаполненияРеквизитов`.
- Табличная часть условий по назначению платежа:
  - `ВидУсловия`: содержит / не содержит
  - `СтрокаПоиска`
  - `БезУчетаРегистра`

### Режимы поиска счета

Для первой версии реализовать:

- `КонкретныйСчет` - пользователь выбирает `Документ.СчетНаОплатуПокупателю`.
- `ПоНомеруВНазначении` - поиск счета по номеру из назначения платежа.
- `ПоКонтрагентуЕслиОдин` - поиск неоплаченного счета по контрагенту/договору, только если найден один надежный вариант.

Заложить на будущее, но не реализовывать в MVP:

- `РаспределитьПоНесколькимСчетам`.

### Режимы заполнения

Реализовать:

- `ТолькоЕслиПусто` - не изменять счет, если типовая БП уже его нашла.
- `Перезаписывать` - заменять счет, найденный типовой БП.

Эти режимы теперь используются не только для счета на оплату, но и для остальных заполняемых реквизитов.

## Технический подход

Основной вариант внедрения:

1. Выполнить штатное распознавание выписки типовой логикой.
2. После штатного распознавания и до создания/сериализации автозаполненного документа применить правила расширения.
3. Для подходящего правила добавить или заменить в `РаспознаннаяОперация.ХозяйственнаяОперация` свойство `СчетНаОплату`.
4. Дальше типовая логика сама перенесет `СчетНаОплату` в `РасшифровкаПлатежа`.

Предпочтительная точка внедрения:

- После формирования `РаспознаннаяОперация.ХозяйственнаяОперация` в процессе `РаспознатьВыписки()`.
- До вызова `СохранитьДанныеАвтозаполненияСтрокВыписки()`.

Причина:

- В этом месте еще нет записанного документа, но уже есть контрагент, договор, вид операции, сумма и реквизиты банковской операции.
- Сериализованные данные автозаполнения будут уже содержать результат правила.

## Открытые технические решения

- Выбрать точный тип объекта для хранения правил: справочник или регистр сведений.
- Проверить, какие объекты расширения удобнее выгружать/загружать в текущей базе.
- Уточнить доступность заимствования и переопределения нужной процедуры общего модуля в расширении.
- Решить, нужен ли отдельный интерфейс команды открытия правил в разделе `Банк и касса`.

## Не входит в MVP

- Полноценная автоматическая классификация списаний поставщикам и подбор `Документ.СчетОтПоставщика`.
- Автоматическое распределение одного платежа по нескольким счетам.
- Регулярные выражения в условиях назначения платежа.
- Обработка уже найденных ранее банковских документов или документов, созданных на основании платежных поручений.
